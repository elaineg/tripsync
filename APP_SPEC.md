# TripSync
Purpose: For two friends planning a trip or a visit, TripSync turns a pasted day-by-day itinerary into a shared, no-login VISUAL hourly calendar that both can open and edit by link and one-click add to their own Google Calendar.

Problem: Computer-workers planning a multi-day trip or a friend's visit hit the "itinerary handoff": one person drafts a day-by-day plan (Notion / Google Doc) and shares it with a co-traveler/host who needs to SEE it, react to it, and get it into their own calendar. The doc shares with no login but isn't visual or glanceable, so on mobile the co-traveler never opens it (the Emily failure), and it can't push events into a personal calendar (the Joanne failure). Hit every time a trip/visit is planned (knowledge workers: several times a year, opened DAILY on the phone once a trip is active).

Beats alternative: The Notion/Google-Doc handoff (best no-login alternative) is editable and link-shareable but not visual/glanceable and can't push events into a calendar; Google Calendar is visual+hourly+.ics but account-gated, no open-by-link editing, RSVP-only not collaborative proposing. TripSync occupies the unoccupied intersection no incumbent has more than two of: trip content + HOURLY visual day grid + no-login EDIT-by-link + named proposed->confirmed status + clean no-auth Google Calendar export. The sharpest unoccupied piece — the reason this app exists — is PASTE a doc itinerary -> auto-parsed visual calendar: it saves ~15-25 min of re-entering ~20 events by hand on the first stab, and no named competitor auto-parses pasted itinerary text.

Core flows:
1. Create + name trip with TWO equal start options -> secret link. Landing page: "Name your trip" (e.g. "Joanne visits — July") then TWO visibly-equal, side-by-side choices: (A) "Paste an itinerary" -> seeds via the parser (flow 2, unchanged); (B) "Start from a blank calendar" -> creates an empty trip and drops the creator straight onto the calendar at `/t/<secret>` ready to build events directly on the grid (flow 3). Server (Turso) mints a secret id; the link is shown with the honest framing "Anyone with this link can view and edit — share only with your travel companions." localStorage remembers it under "Recent trips" (device-local, no email/account). Reuses the utm-grid Team Workspace pattern (secret-link, last-write-wins, no auth).
2. PASTE-IMPORT (a seeding flow, the differentiator — UNCHANGED). A "Paste itinerary" box accepts day-headed, time-prefixed text. The parser detects DAY HEADERS ("Fri May 1", "Saturday May 2") and time-prefixed lines under each — single times ("12:30PM Emily lands", "9AM: Wake up") AND ranges ("1-2PM Uber", "2-4PM unpack") — stripping the leading time as start (and end for ranges), the rest as the title, preserving any URL as a clickable link. Non-time lines BEFORE the first day header (the "Details" preamble) route to the Trip Details panel. A PREVIEW/CONFIRM screen shows every parsed event bucketed by day before committing. A "Load sample itinerary" button fills the box with the Emily acceptance-test itinerary so anyone can try it cold.
3. BUILD ON THE GRID — Google-Calendar-style direct creation (the blank-calendar path's primary interaction). DESKTOP PRIMARY: on the week/day-hourly grid, press-and-drag from a start time to an end time, release -> an event spanning that exact duration is created, then an inline quick-create title/details popover (GCal quick-create). A single click without dragging = a default ~1-hour block you can then drag-resize. Existing events: drag the body to MOVE, drag a top/bottom edge to RESIZE (15-min snap). Month view: click a day to create on it / jump to its day view. The "+ Add event" button is ONLY a secondary mobile/touch FALLBACK — it is NOT the primary desktop path. Direct-created events are written with the same `TripEvent` shape as parsed ones (proposed status, author color, add-to-Google-Calendar, .ics — identical behavior). Day is the HERO view (mobile-glanceable, 15-min snap); a pinned Trip Details panel sits at top for non-event notes. The mobile day grid is scroll-safe: a plain vertical swipe always scrolls; on TOUCH, drag-to-create is disabled (When2meet's failure mode) — touch uses tap-an-empty-slot-to-create-a-default-block + the "+ Add event" button; moving/resizing an existing event requires an explicit drag-handle or long-press (never hijacks vertical scroll).
4. Open by link, collaborate, and export (no login). A friend opens `/t/<secret>`, browses first, types their NAME on first interaction (localStorage keyed by the secret + a participantId UUID; same-device return auto-resumes). They light-edit / add (via flow 3) / delete events, and CONFIRM an event — confirming attributes by name ("Confirmed by Joanne") and flips proposed->confirmed (dashed/translucent -> solid-with-check). No per-event ownership locks; the link is the boundary. Writes are debounced (~2-3s idle, flushed on blur/visibilitychange via sendBeacon, event-level rows, soft-delete tombstones); refresh on tab-focus + manual Refresh (no realtime polling). EXPORT: per-event "Add to calendar" -> Google Calendar render URL (`calendar.google.com/calendar/render?action=TEMPLATE&text=...&dates=...&ctz=...`, no OAuth/API) or .ics; bulk "Save to calendar (.ics)" downloads every confirmed event.

Success checks:
1. EMILY ACCEPTANCE TEST (named, seedable): clicking "Load sample itinerary" then "Parse" then "Confirm" produces, on the calendar: Fri May 1 events at 12:30PM (Emily lands), 1-2PM (Uber to 123 Main St), 2-4PM (unpack), 4:30PM (walk to Dolores/Empress Vintage), 5:15PM (dinner Foreign Cinema), 7:30PM (20spot/arcana walk), 8:30PM (El Chato); and Sat May 2 events at 9AM, 11AM (brunch That's My Jam), 12PM (Haight vintage), 6PM (dinner Bansang), 10PM (Bar Part Time). Both single times ("12:30PM") AND ranges ("1-2PM") parse; both "9AM:" and "12:30PM" formats parse.
2. The Details preamble (weather 10-20deg, bring light jacket/cardigan, skirt ok, bring ID) appears in the Trip Details panel and NOT as calendar events.
3. The Partiful URL in the El Chato line is preserved and rendered as a clickable `<a>` link (href starts with http), tappable on the event detail.
4. A preview/confirm step is shown before the parse commits: the parsed events are visible and dismissable BEFORE any event is written to the server.
5. Creating a trip yields a URL of the form `/t/<secret>`; opening that URL in a fresh browser (no localStorage) loads the same trip and its events without any login prompt.
6. The invite/share UI states verbatim that anyone with the link can view and edit (honest secret-link framing); no email, password, or account field appears anywhere in the app.
7. TWO START OPTIONS: the landing/create flow shows two visibly-equal start choices — "Paste an itinerary" AND "Start from a blank calendar" — rendered with equal weight (neither hidden behind a menu or styled as the lone primary button).
8. BLANK-CALENDAR PATH: choosing "Start from a blank calendar" mints a `/t/<secret>` trip with ZERO events and lands the user directly on the calendar grid (the paste panel does NOT auto-take-over the screen); the grid is ready for direct event creation.
9. DESKTOP DRAG-CREATE: on a desktop (mouse) viewport, pressing on the day-hourly grid at 9:00 and dragging to 11:00 then releasing creates a single event with startMinutes=540 and endMinutes=660 (a 2-hour block), and an inline quick-create title popover appears for that new event.
10. DESKTOP CLICK-CREATE: a single click on an empty grid slot (no drag) creates a default ~1-hour block (endMinutes = startMinutes + 60) that can then be drag-resized.
11. MOVE / RESIZE: dragging an existing event's body to a new time changes its start/end by the dragged amount (15-min snap); dragging its top or bottom edge changes only that edge (resize), leaving the other edge fixed.
12. DIRECT EVENT == PARSED EVENT: an event created by drag-on-grid has status "proposed" by default, exposes the same per-event "Add to Google Calendar" action, and is included in the bulk "Save to calendar (.ics)" download (appears as a VEVENT) — indistinguishable in behavior from a parsed event.
13. MOBILE/TOUCH (≈390px wide): drag-to-create is NOT active; a tap on an empty slot creates a default 1-hour block AND a "+ Add event" button is present as the touch create path. A plain vertical swipe over the grid SCROLLS it (does not create or move an event); an existing event only moves via its drag-handle or after a long-press. (Verified on a real touch viewport, not desktop drag.)
14. PARSER UNCHANGED: the EMILY ACCEPTANCE TEST (check 1) still passes verbatim — "Load sample itinerary" -> "Parse" -> "Confirm" produces the same events with the same single-time and range parsing; the paste flow is reachable from both the landing "Paste an itinerary" option and the in-trip "Paste itinerary" link.
15. Confirming an event after typing the name "Joanne" shows "Confirmed by Joanne" and flips the event from dashed/translucent (proposed) to solid-with-check (confirmed).
16. Per-event "Add to Google Calendar" opens a `calendar.google.com/calendar/render?action=TEMPLATE` URL whose `dates` and `text` match the event; "Save to calendar (.ics)" downloads a .ics file containing exactly the confirmed events (correct VEVENT count and DTSTART times).
17. First-persist correctness: opening an existing trip in a fresh browser does NOT overwrite server state with an empty calendar before hydration completes (load-on-open read succeeds before any write fires).

Out of scope: live realtime/websocket sync (refresh-on-focus only); field-level merge (event-level last-write-wins + soft-delete tombstones); accounts, login, passwords, email recovery; subscribe-by-URL live calendar feed; expenses, maps, place search, flight tracking; per-event ownership/permission locks; AI/LLM parsing (deterministic rule-based parser only); RECURRING/repeating events; a drag that spans across midnight into the next day (a single event stays within one calendar day; create separate events per day); multi-day click-drag in month view (month-day click opens/creates within that one day only); any paid API, OAuth, or paid infra.
