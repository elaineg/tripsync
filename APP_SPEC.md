# TripSync
Purpose: For two friends planning a trip or a visit, TripSync turns a pasted day-by-day itinerary into a shared, no-login VISUAL hourly calendar that both can open and edit by link and one-click add to their own Google Calendar.

Problem: Computer-workers planning a multi-day trip or a friend's visit hit the "itinerary handoff": one person drafts a day-by-day plan (Notion / Google Doc) and shares it with a co-traveler/host who needs to SEE it, react to it, and get it into their own calendar. The doc shares with no login but isn't visual or glanceable, so on mobile the co-traveler never opens it (the Emily failure), and it can't push events into a personal calendar (the Joanne failure). Hit every time a trip/visit is planned (knowledge workers: several times a year, opened DAILY on the phone once a trip is active).

Beats alternative: The Notion/Google-Doc handoff (best no-login alternative) is editable and link-shareable but not visual/glanceable and can't push events into a calendar; Google Calendar is visual+hourly+.ics but account-gated, no open-by-link editing, RSVP-only not collaborative proposing. TripSync occupies the unoccupied intersection no incumbent has more than two of: trip content + HOURLY visual day grid + no-login EDIT-by-link + named proposed->confirmed status + clean no-auth Google Calendar export. The sharpest unoccupied piece — the reason this app exists — is PASTE a doc itinerary -> auto-parsed visual calendar: it saves ~15-25 min of re-entering ~20 events by hand on the first stab, and no named competitor auto-parses pasted itinerary text.

Core flows:
1. Create + name trip -> secret link. Landing page: "Name your trip" (e.g. "Joanne visits — July") -> server (Turso) mints a secret id -> creator lands on the calendar at `/t/<secret>`. "Copy invite link" button; the link is shown with the honest framing "Anyone with this link can view and edit — share only with your travel companions." localStorage remembers it under "Recent trips" (device-local, no email/account). Reuses the utm-grid Team Workspace pattern (secret-link, last-write-wins, no auth).
2. PASTE-IMPORT (PRIMARY seeding flow, the differentiator). A "Paste itinerary" box accepts day-headed, time-prefixed text. The parser detects DAY HEADERS ("Fri May 1", "Saturday May 2") and time-prefixed lines under each — single times ("12:30PM Emily lands", "9AM: Wake up") AND ranges ("1-2PM Uber", "2-4PM unpack") — stripping the leading time as start (and end for ranges), the rest as the title, preserving any URL as a clickable link. Non-time lines BEFORE the first day header (the "Details" preamble) route to the Trip Details panel. A PREVIEW/CONFIRM screen shows every parsed event bucketed by day before committing. A "Load sample itinerary" button fills the box with the Emily acceptance-test itinerary so anyone can try it cold.
3. Visual month / week / DAY-HOURLY views (DAY is the HERO, mobile-glanceable, 15-min snap). A pinned Trip Details panel sits at the top for non-event notes (weather, what to bring, dress code). Events render with author color and clickable links in title/notes; proposed events are dashed/translucent, confirmed are solid with a check. The mobile day grid is scroll-safe: a plain vertical swipe always scrolls; move/resize requires an explicit drag-handle or long-press (never hijacks vertical scroll).
4. Open by link + collaborate (no login). A friend opens `/t/<secret>`, browses first, and types their NAME on first interaction (stored in localStorage keyed by the secret with a participantId UUID; returning on the same device auto-resumes). They can light-edit / add / delete events, and CONFIRM an event — confirming attributes by name ("Confirmed by Joanne") and flips proposed->confirmed. No per-event ownership locks; the link is the boundary. Writes are debounced (~2-3s idle, flushed on blur/visibilitychange via sendBeacon, event-level rows, soft-delete tombstones); refresh on tab-focus + a manual Refresh button (no realtime polling).
5. Add-to-Google-Calendar. Per-event "Add to calendar" -> Google Calendar render URL (`calendar.google.com/calendar/render?action=TEMPLATE&text=...&dates=...&ctz=...`, no OAuth, no API) or download .ics. Bulk "Add all confirmed to my calendar" -> one .ics download of every CONFIRMED event.

Success checks:
1. EMILY ACCEPTANCE TEST (named, seedable): clicking "Load sample itinerary" then "Parse" then "Confirm" produces, on the calendar: Fri May 1 events at 12:30PM (Emily lands), 1-2PM (Uber to 123 Main St), 2-4PM (unpack), 4:30PM (walk to Dolores/Empress Vintage), 5:15PM (dinner Foreign Cinema), 7:30PM (20spot/arcana walk), 8:30PM (El Chato); and Sat May 2 events at 9AM, 11AM (brunch That's My Jam), 12PM (Haight vintage), 6PM (dinner Bansang), 10PM (Bar Part Time). Both single times ("12:30PM") AND ranges ("1-2PM") parse; both "9AM:" and "12:30PM" formats parse.
2. The Details preamble (weather 10-20deg, bring light jacket/cardigan, skirt ok, bring ID) appears in the Trip Details panel and NOT as calendar events.
3. The Partiful URL in the El Chato line is preserved and rendered as a clickable `<a>` link (href starts with http), tappable on the event detail.
4. A preview/confirm step is shown before the parse commits: the parsed events are visible and dismissable BEFORE any event is written to the server.
5. Creating a trip yields a URL of the form `/t/<secret>`; opening that URL in a fresh browser (no localStorage) loads the same trip and its events without any login prompt.
6. The invite/share UI states verbatim that anyone with the link can view and edit (honest secret-link framing); no email, password, or account field appears anywhere in the app.
7. On a touch viewport (e.g. 390px wide), a plain vertical swipe over the day grid SCROLLS the grid (does not move an event); an event only moves via its drag-handle or after a long-press. (Verified on a real touch viewport, not desktop drag.)
8. Confirming an event after typing the name "Joanne" shows "Confirmed by Joanne" and flips the event from dashed/translucent (proposed) to solid-with-check (confirmed).
9. Per-event "Add to Google Calendar" opens a `calendar.google.com/calendar/render?action=TEMPLATE` URL whose `dates` and `text` match the event; "Add all confirmed" downloads a .ics file containing exactly the confirmed events (correct VEVENT count and DTSTART times).
10. First-persist correctness: opening an existing trip in a fresh browser does NOT overwrite server state with an empty calendar before hydration completes (load-on-open read succeeds before any write fires).

Out of scope: live realtime/websocket sync (refresh-on-focus only); field-level merge (event-level last-write-wins + soft-delete tombstones); accounts, login, passwords, email recovery; subscribe-by-URL live calendar feed; expenses, maps, place search, flight tracking; per-event ownership/permission locks; AI/LLM parsing (deterministic rule-based parser only); any paid API, OAuth, or paid infra.
